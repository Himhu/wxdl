### 2. 前端站点切换后数据不同步 ✅ 已修复

**发现者**: Gemini（前端架构专家）
**修复日期**: 2026-03-19
**修复者**: Claude + Gemini 协作

**问题描述**:
- 用户在工作台切换站点（A → B）后
- 切换到"卡密"Tab 时，列表显示的仍是站点 A 的数据
- 但此时 API 请求已经携带站点 B 的 `X-Site-Id`
- 导致数据错乱和潜在的跨站点操作

**根本原因**:
- `components/station-switch/index.js` 切换站点时更新了 `stationStore.currentSite`
- 但页面没有监听 `currentSite` 变化，不会自动刷新数据
- `onShow` 生命周期中也没有检查站点是否变化

**已实施的修复**:
1. ✅ 修复 `pages/tabbar/cards/index.js` - 卡密列表页
2. ✅ 修复 `pages/logs/index.js` - 操作日志页
3. ⏭️ 跳过 `pages/finance/records/index.js` - 空占位页面
4. ⏭️ 跳过 `pages/proxy/list/index.js` - 空占位页面

**修复方案（Gemini 设计）**:
- 使用手动 `onShow()` 检查而非 MobX bindings
- 在 `onLoad()` 中记录 `this.lastLoadedSite`
- 在 `onShow()` 中检测站点变化
- 站点变化时：重置分页、清空列表、重新加载

**性能优化**:
- 只在站点真正变化时重新加载（避免每次 `onShow` 都加载）
- 防止用户查看详情返回时丢失滚动位置
- 清空旧数据避免 UI 闪烁

**优先级**: P0 - 已完成

### 3. 数据库 Schema 管理混乱 ✅ 已修复

**发现者**: Codex（后端架构专家）
**修复日期**: 2026-03-19
**修复者**: Claude + Gemini 协作

**问题描述**:
- 项目同时使用 `AutoMigrate` 和 SQL 脚本管理 Schema
- 两种方式管理的表不一致，导致部署时容易遗漏

**已实施的修复**:

1. ✅ **引入 golang-migrate** - 版本化数据库迁移
   - 新增 `backend/internal/database/migrate.go` - 迁移运行器（启动时自动执行）
   - 新增 `backend/internal/database/migrations/000001_baseline.up.sql` - 统一基线（12个表）
   - 新增 `backend/internal/database/migrations/000001_baseline.down.sql` - 回滚脚本

2. ✅ **移除 AutoMigrate** - `main.go` 改用 `database.RunMigrations()`

3. ✅ **补全缺失的 stations 表** - 在基线迁移中创建

4. ✅ **移除未使用的表** - `wechat_bindings` 和 `operation_logs` 不再包含

5. ✅ **添加外键约束**:
   - `agents.parent_id` → `agents.id` (ON DELETE SET NULL)
   - `cards.agent_id` → `agents.id` (ON DELETE RESTRICT)
   - `balance_logs.agent_id` → `agents.id` (ON DELETE RESTRICT)
   - `recharge_requests.agent_id` → `agents.id` (ON DELETE RESTRICT)
   - `system_setting_revisions.setting_id` → `system_settings.id` (ON DELETE CASCADE)
   - `admin_users.role_id` → `admin_roles.id` (ON DELETE RESTRICT)
   - `admin_role_permissions.role_id` → `admin_roles.id` (ON DELETE CASCADE)

6. ✅ **统一 3 个仅 AutoMigrate 的表** - mini_program_config_items, system_settings, system_setting_revisions 纳入基线 SQL

7. ✅ **归档旧 SQL 脚本** - 移至 `scripts/archived/`

8. ✅ **DSN 添加 multiStatements=true** - golang-migrate 必需参数

**优先级**: P0 - 已完成

### 4. 微信账号绑定存在并发竞态

**发现者**: Codex

**问题描述**:
- 两个账号同时绑定同一个 `openid` 时
- 检查逻辑存在竞态窗口（check-then-act）
- 虽然数据库唯一索引会保护，但错误码不正确

**代码位置**:
```go
// backend/internal/service/auth_service.go:168
existingAgent, err := s.agentRepository.GetByWechatOpenID(ctx, session.OpenID)
if existingAgent != nil && existingAgent.ID != agent.ID {
    return nil, utils.ConflictError("wechat account already bound")
}
// 竞态窗口：另一个请求可能在这里插入相同的 openid
if err := s.agentRepository.BindWechat(ctx, agent.ID, session.OpenID, session.UnionID); err != nil {
    return nil, utils.InternalError(err) // 应该返回 409 而非 500
}
```

**修复方案**:
```go
if err := s.agentRepository.BindWechat(ctx, agent.ID, session.OpenID, session.UnionID); err != nil {
    if isDuplicateKeyError(err) {
        return nil, utils.ConflictError("wechat account already bound")
    }
    return nil, utils.InternalError(err)
}
```

**优先级**: P1
